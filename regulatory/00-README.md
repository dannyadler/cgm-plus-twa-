## Purpose

This artifact set demonstrates an end-to-end DHF-style document collection generated by AI from the source code of the Trinity CGM Plus presale demo PWA. It accompanies a customer-facing presentation on AI-assisted regulatory documentation.

The pitch:

- **AI generates the foundational drafts in minutes** — risk analysis, requirements spec, design description, test plan, cybersecurity assessment, SBOM, traceability matrix.
- **Human RA/QA review and validate** — clinical severity calibration, QMS integration, sign-offs.
- **Tests are real** — the test specifications below are wired to a CI pipeline that runs on every code push, producing actual V&V evidence.
- **Documents stay current** — code change → CI run → SBOM regenerated, traceability re-checked, test outcomes refreshed.

## Honest framing

Three rules the AI was asked to hold:

1. **Every document carries an AI-generated-draft disclaimer.** Front matter on every document calls out that human review is required. The pitch only works if the framing is honest — regulatory-savvy reviewers spot fabricated DHFs immediately.

2. **Tests are real, not theatrical.** Every test in the STD has a corresponding implementation (or an explicit "recommended addition" note). The CI pipeline executes them. There are no fabricated test reports.

3. **Traceability is wired up.** Every requirement, design item, risk, and test cross-references by ID. The traceability matrix exposes coverage gaps rather than hiding them.

## What AI does well

- **Document scaffolding** — pulling structure from standards (IEC 62304, ISO 14971, FDA cybersecurity guidance), organizing requirements, generating tables.
- **Cross-referencing** — keeping REQ-IDs / DES-IDs / RISK-IDs / TST-IDs consistent across nine documents.
- **Code-grounded honesty** — flagging real bugs in the codebase as risks, real gaps as open items, real coverage holes in the matrix.
- **Drafting velocity** — order-of-magnitude faster than manual authorship for foundational drafting.

## What AI does NOT replace

- **Clinical severity calibration.** A clinical reviewer must validate severity scores against published harm taxonomies (MAUDE, IMDRF terminology).
- **Use-error analysis (IEC 62366).** Formative usability requires real users.
- **QMS integration.** These documents follow standard structures, not Trinity Biotech's quality manual templates, document numbering, or approval workflow.
- **Sign-off and accountability.** Approval blocks are placeholder. Real signatures represent real accountability under the QMS.
- **Penetration testing.** AI can write a threat model. It cannot probe a deployed system.
- **SOUP qualification per IEC 62304 §8.1.2.** Components are listed, but formal qualification records require human evaluation.

## Evaluation prompts for human reviewers

Pick a document and answer these:

1. Is the technical content **correct** with respect to the actual codebase?
2. Is the regulatory **structure and terminology** appropriate?
3. Are there **claims that should be flagged** as needing more rigor or evidence?
4. What's **missing** that a Class B (or Class C) submission would require?
5. Could this draft accelerate your team's work, or is the rework cost higher than starting from scratch?

Send feedback to the project owner; iterations on the AI prompt and document templates are cheap.
